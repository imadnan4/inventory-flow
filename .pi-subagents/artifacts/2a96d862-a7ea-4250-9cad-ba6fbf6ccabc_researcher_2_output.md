# Research: EF Core / SQL Server inventory receipt-and-adjustment ledger

## Summary
Implement each receipt/adjustment as one short database transaction that inserts an immutable movement event, records a durable idempotency key, and conditionally updates the inventory-balance projection. Use SQL Server `rowversion` as EF Core's optimistic-concurrency token and treat a zero-row update / `DbUpdateConcurrencyException` as a retry-or-conflict outcome, never as a successful receipt. Explicitly define compatible `decimal(p,s)` types for quantities and costs, and make the event stream append-only.

## Findings
1. **Make the event, deduplication record, and balance change one commit unit (blocker if split).** EF Core guarantees a transaction for one `SaveChanges`; use an explicit transaction when the operation spans multiple saves/commands, committing only after all three writes succeed. With SQL retry enabled, execute the *entire* explicit transaction through `CreateExecutionStrategy`, rather than retrying individual statements. [EF Core transactions](https://learn.microsoft.com/en-us/ef/core/saving/transactions) · [EF Core connection resiliency](https://learn.microsoft.com/en-us/ef/core/miscellaneous/connection-resiliency)
   - Recommendation: transaction steps: (a) insert `InventoryEvent` (receipt/adjustment delta and immutable business facts), (b) insert an `InventoryOperation`/inbox row keyed by a client-supplied request GUID, (c) conditionally update/insert the balance projection, then commit. Roll back all on duplicate/conflict/error. Keep it short; do not include external I/O.
2. **Use `rowversion` on the mutable balance projection, not as an event identity (high).** Map `byte[] Version` with `[Timestamp]` or `.IsRowVersion()`. EF emits an `UPDATE ... WHERE [key] = @key AND [Version] = @originalVersion`; a concurrent change produces zero affected rows and `DbUpdateConcurrencyException`. SQL Server says `rowversion` is an 8-byte, database-relative increment—not a time—and warns it is a poor key because every update changes it. [EF Core concurrency](https://learn.microsoft.com/en-us/ef/core/saving/concurrency) · [SQL Server rowversion](https://learn.microsoft.com/en-us/sql/t-sql/data-types/rowversion-transact-sql?view=sql-server-ver17)
   - Recommendation: on conflict, discard/reload the balance and retry the complete idempotent operation (or return a conflict); do not write the ledger event independently. If issuing SQL directly, include `WHERE BalanceId=@id AND RowVersion=@original` and require exactly one row updated; SQL Server documents this conditional pattern.
3. **Idempotency must be enforced by a database uniqueness boundary (blocker if client-only).** Put a non-null client-generated operation/request ID on the event or a companion operation table and enforce `UNIQUE` (normally scoped to tenant/source as appropriate). SQL Server enforces a `UNIQUE` constraint through a unique index; duplicate inserts fail and no duplicate row is added. This handles retries and concurrent duplicate submissions. [SQL Server unique constraints](https://learn.microsoft.com/en-us/sql/relational-databases/tables/unique-constraints-and-check-constraints?view=sql-server-ver17)
   - Recommendation: on a duplicate-key exception, read the existing completed event/result and return it; never apply the delta again. This also covers an unknown commit outcome: EF Core documents that a dropped connection during commit leaves outcome unknown and retries can duplicate/corrupt non-idempotent writes. Prefer client GUIDs and/or `ExecuteInTransaction(... verifySucceeded: ...)`. [EF Core commit idempotency guidance](https://learn.microsoft.com/en-us/ef/core/miscellaneous/connection-resiliency)
4. **Declare precision/scale deliberately and validate domain bounds (high).** EF Core can map `[Precision(14,2)]` / `.HasPrecision(p,s)` to SQL Server `decimal(p,s)`; precision is total digits and scale is fractional digits. EF does not validate precision or scale before submission, so database constraints/application validation must reject invalid quantities/costs. [EF Core entity properties](https://learn.microsoft.com/en-us/ef/core/modeling/entity-properties)
   - Recommendation: choose `p,s` from maximum stock/cost and smallest supported unit (for example, quantity `decimal(19,6)` only if six fractional units are required); use the same scale for event delta and projected balance, and define monetary/unit-cost precision separately. Do not rely on defaults or floating point. SQL Server supports maximum decimal precision 38 and may round conversion to lower scale; arithmetic can reduce scale/overflow when precision exceeds 38. [SQL Server decimal/numeric](https://learn.microsoft.com/en-us/sql/t-sql/data-types/decimal-and-numeric-transact-sql?view=sql-server-ver17)
5. **Keep audit events append-only; use SQL Ledger only when tamper-evidence is required (medium).** Model corrections as compensating adjustment events, not `UPDATE`/`DELETE` of prior events. On SQL Server 2022+/Azure SQL, append-only ledger tables block updates/deletes and provide tamper-evidence; cryptographic assurance additionally depends on generating and retaining database digests outside the database in immutable storage. [SQL Server Ledger overview](https://learn.microsoft.com/en-us/sql/relational-databases/security/ledger/ledger-overview?view=sql-server-ver17)
   - Recommendation: an append-only `InventoryEvent` is the audit source of truth; `InventoryBalance` is a replaceable mutable projection. If using an updatable ledger table for balances, it retains history but is not an immutable event stream; its ledger view supplies transaction and operation metadata. [Updatable ledger tables](https://learn.microsoft.com/en-us/sql/relational-databases/security/ledger/ledger-updatable-ledger-tables?view=sql-server-ver17)

## Gaps
- No domain limits were supplied, so exact quantity and monetary `decimal(p,s)` values cannot be prescribed.
- SQL Server Ledger requires SQL Server 2022+ or applicable Azure SQL; otherwise use append-only permissions/schema plus independent audit retention, without claiming Ledger-grade tamper evidence.
- Verify the actual migration contains the unique operation key, rowversion mapping, decimal definitions, and tests for duplicate replay, concurrent receipt, and unknown-commit recovery.
