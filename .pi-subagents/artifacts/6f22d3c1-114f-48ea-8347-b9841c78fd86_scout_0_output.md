## Key Code

### Recommendation — P1 product catalog: list and create, with archive (not edit, categories, stock, or warehouses)
Deliver one authenticated **Products** page where a signed-in user can create a product, see their active product list, and archive a product. This is the smallest shippable inventory-management slice: it creates durable inventory identity and immediately makes the existing Products navigation useful. Include archive because it gives a safe lifecycle without hard deletion; defer edit because create/list/archive already verifies identity, persistence, authorization, query, and UI mutation paths.

**Initial model:** `Product(Id, OwnerUserId, Sku, Name, UnitOfMeasure, IsArchived, CreatedAtUtc, ArchivedAtUtc)`.

* Keep values deliberately narrow: required SKU/name, a small required unit string (for example `each`), UTC timestamps, and archive state. Do **not** introduce price, on-hand quantity, reorder level, category, supplier, warehouse, tax/currency, or movement ledger in this slice.
* The model belongs in the Domain project; EF mapping and query implementation belong in Infrastructure; commands/queries, DTOs, and validation belong in Application; extracting the user ID and translating to HTTP belongs in API.

### Ownership boundary (mandatory constraint)
There is no workspace/tenant aggregate or membership table (`ApplicationDbContext.cs:19-29`; `ApplicationUser.cs:9-15`). Therefore the shippable ownership boundary must be **`OwnerUserId` (the JWT NameIdentifier), server assigned**:

* Never accept `OwnerUserId` from JSON, route, query, or UI state.
* Every list, lookup, and archive operation filters `OwnerUserId == authenticatedUserId`; archive must not reveal whether a foreign product exists (return 404 after the scoped lookup).
* SKU uniqueness is `(OwnerUserId, normalized SKU)` among active products. Normalize/validate consistently (trim; choose and document case-insensitive matching through a normalized persisted value or SQL Server collation). A global SKU unique index is incorrect because different users may legitimately use the same SKU.
* This is a conscious temporary single-user ownership model, not a claim that the SaaS already has workspace isolation. Add `Workspace`/membership before shared-user catalog access; migration to `(WorkspaceId, NormalizedSku)` will be required then. Do not add a fake default workspace in this slice—it adds unactionable membership/authorization behavior.

### API acceptance criteria
1. `GET /api/products` requires bearer authentication and returns `200` with only the caller's non-archived products, sorted deterministically (for example `Name`, then `Id`). Unauthenticated is `401`.
2. `POST /api/products` requires bearer authentication; accepts only `{ sku, name, unitOfMeasure }`; returns `201 Created` with the server-generated ID and product representation plus a `Location` header. The server derives owner from the JWT.
3. Invalid/blank/over-length fields yield the existing RFC 7807 validation `400` shape (`GlobalExceptionHandler.cs:29-57`). Duplicate active SKU for that owner is a stable client error (`409 Conflict` preferred; add an explicit exception mapping rather than leaking SQL errors).
4. `DELETE /api/products/{id}` (archive semantic) requires bearer auth and returns `204`; it sets archive state/timestamp rather than deleting. The item disappears from the default list. A foreign or nonexistent ID returns `404`; retrying an already archived own item may return `404` (document this idempotency choice).
5. A second authenticated user cannot list, archive, or conflict with the first user's SKU. The same normalized SKU is allowed across different owners.

### UI acceptance criteria
1. `/products` remains behind `RequireAuth` and becomes a real page, while the sidebar route remains unchanged.
2. Initial loading, empty, error, and populated states are visible. Empty state explains that no products exist and exposes the create action.
3. The page has an accessible create form/dialog with labels and inline validation for SKU, name, and unit. Submit is disabled while pending; successful creation refreshes/inserts the list and displays a success notification; API validation errors are rendered without losing entered values.
4. Each row shows SKU, name, unit, and an Archive action. Confirmation prevents accidental archive; success removes it from the active list. All requests use `apiClient`, not a bespoke token mechanism, preserving current refresh/retry behavior.

## Architecture
The browser restores an auth session and `apiClient` supplies the bearer token (`frontend/src/lib/api-client.ts:19-57`). Router guards protect the Products URL (`frontend/src/app/router/index.tsx:30-53`). `ProductsPage` calls its feature API module, whose requests travel through the existing API client. `ProductsController` reads the signed JWT subject as an internal user scope, sends product commands/queries through already assembly-scanned MediatR, and returns DTOs. Infrastructure persists the Domain product through EF Core/SQL Server, including an owner-scoped SKU constraint. Integration tests should authenticate through the actual registration endpoint and exercise the complete route/EF/migration path.

This is preferable to alternatives:
* **Inventory movements/warehouse quantities:** require a settled product identity plus warehouse/location and immutable ledger/concurrency semantics; much larger and unsafe without the catalog root.
* **Categories/suppliers:** supporting reference data alone has less direct operational value and still leaves no item to classify/source.
* **Purchase/sales orders:** require products and likely counterparties, order lifecycle, quantities, and stock side effects.
* **Dashboard integration:** `DashboardPage.tsx:28-94` is demo presentation; product count alone would create misleading mixed real/demo operational metrics.
* **Roles/workspaces first:** valuable later, but no current membership schema or product record exists. User-scoped catalog is a real, contained workflow; shared workspace semantics should be designed deliberately, not inferred from authentication.

## Review Findings / Residual Risks
* **High — workspace isolation is not implemented:** `backend/src/InventoryFlow.Infrastructure/Persistence/ApplicationDbContext.cs:19-29` and `ApplicationUser.cs:9-15` contain no workspace/membership ownership data. Treat this slice as private user catalogs only; do not market multi-user workspace sharing until a workspace migration and authorization policy exist.
* **High — ownership must be server-side:** `AuthController.cs:63-72` demonstrates the valid principal source. Any request DTO containing owner/workspace IDs would permit horizontal data access; do not add one.
* **Medium — unique SKU behavior needs an explicit normalization decision:** SQL Server string comparison/collation may otherwise make `abc` and `ABC` behavior environment-dependent. Persist a normalized SKU and constrain it, or explicitly configure a known case-insensitive comparison.
* **Medium — database duplicate violations cannot be allowed to become 500:** `GlobalExceptionHandler.cs:29-57` currently maps only validation/domain/auth exceptions. Translate the known unique-constraint violation to a stable conflict response.
* **Low — static dashboard data remains misleading if presented as live:** `frontend/src/features/dashboard/pages/DashboardPage.tsx:28-94` must remain clearly demo/static until a full dashboard slice replaces all related metrics.
