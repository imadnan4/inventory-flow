### Prioritized recommendation
**P0 — Deliver authenticated workspace access: register, login, refresh-token rotation, logout, and protected SPA shell.**

This is the next smallest viable vertical slice. It turns the existing Identity and refresh-token persistence foundations into a user-facing, secure workspace boundary, makes the dashboard an authenticated destination rather than publicly exposed mock UI, and unblocks the next P1 catalog slice without retrofitting authorization/session handling. It is explicitly the first upcoming slice in `README.md:139` and requires no new inventory schema.

**P1 immediately after P0 — Product catalog CRUD (list/create/edit/archive) for the signed-in user/workspace.** This is the first direct inventory-record value, but should not precede P0 because the current app has neither authenticated principal nor tenant/workspace model. Avoid warehouses, stock movements, purchasing, sales, and reporting until catalog identity/ownership semantics are settled.

### P0 scope and acceptance criteria
1. `POST /api/auth/register` validates display name, email, and the existing Identity password policy; creates the Identity user and returns a short-lived JWT plus refresh token according to one documented transport choice.
2. `POST /api/auth/login` accepts valid credentials, honors Identity lockout failures, and returns the same session contract. Invalid credentials disclose no account-existence detail.
3. `POST /api/auth/refresh` accepts an active refresh token, atomically revokes it, persists a replacement hashed token, and returns a new access/refresh pair. Reusing a revoked, expired, malformed, or unknown refresh token returns 401 and creates no session.
4. `POST /api/auth/logout` revokes the supplied active refresh token; calling refresh afterward is unauthorized. Refresh tokens are never persisted or logged in plaintext.
5. JWT bearer validation (issuer, audience, signing key, expiry) is configured from non-committed configuration; `UseAuthentication()` precedes `UseAuthorization()`. A small protected `GET /api/auth/me` proves the principal/claims pipeline.
6. SPA supplies Login and Register pages; unauthenticated visitors to `/dashboard` and other workspace routes redirect to login; valid login returns to the requested route. The Axios client attaches the access token and performs one controlled refresh/retry or redirects to login on session expiry.
7. Unit tests cover refresh-token issuance/rotation edge cases; integration tests cover registration, login, protected endpoint 401/200, rotation, reuse rejection, and logout. Frontend typecheck/lint/build remain clean.

### Dependencies / decisions to make in implementation
- Add `Microsoft.AspNetCore.Authentication.JwtBearer`; it is absent from central package versions and API project references.
- Provide `Jwt` settings through environment/secret configuration (signing key must not be committed). CORS needs a real frontend origin: current `Cors:AllowedOrigins` is empty (`appsettings.json:2-4`).
- Choose refresh transport deliberately: secure HttpOnly/SameSite cookie is preferred for browser use; if JSON body/local storage is selected, document the XSS tradeoff and do not persist tokens in Zustand/localStorage by default.
- Define initial registration policy (open self-service versus seeded/invite-only admin). Do **not** add roles/permissions to P0 beyond claims plumbing; roadmap places enforcement after authentication.
- A workspace/tenant ownership model is not present. P1 must first decide whether Product ownership is user- or organization-based; do not bake user-only ownership into a future SaaS multi-tenant schema without a product decision.

## Architecture

`Program.cs` composes Application and Infrastructure, while Infrastructure currently provides Identity's `UserManager`, `SignInManager`, EF store, and role services. The missing link is an Application-owned authentication use case that invokes those services through abstractions and delegates JWT creation/persistence to Infrastructure. The API exposes that contract; the React auth feature calls it through the existing Axios client, owns transient session state, and gates the existing `DashboardLayout` routes. The existing `RefreshToken` object supports the critical rotate/revoke behavior and database indexes support lookup by hash and user/expiry.
