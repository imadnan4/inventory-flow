## Review
- Correct: CORS credential flow is configured on both clients (`frontend/src/lib/api-client.ts:5-14`) and API policy (`backend/src/InventoryFlow.Api/Program.cs:34-40`). Bearer `/me` is protected and tested (`AuthController.cs:63-72`, `AuthenticationEndpointsTests.cs:53-69`). Form labels, autocomplete, invalid state, and alert errors are present (`LoginPage.tsx:45-86`, `RegisterPage.tsx:47-100`).
- Blocker: **High** — Logout can leave a valid rotated refresh cookie/session after an in-flight refresh. Client epoch invalidation prevents Zustand resurrection only (`frontend/src/lib/api-client.ts:24-41`), while refresh still writes its `Set-Cookie` (`AuthController.cs:47-50`). `LogoutAsync` has no transaction/serialization boundary (`IdentityAuthenticationService.cs:90-96`), so it can query/revoke the family before a concurrent refresh commits its descendant; that descendant remains active and its later response can restore the cookie. Smallest fix: perform logout lookup and family revocation in the same retryable serializable transaction as refresh, and add a concurrent refresh/logout integration test asserting every resulting refresh token is unauthorized.
- Blocker: **High** — Requested protected path is lost when a user changes between sign-in and registration. `RequireAuth` stores `from` (`RequireAuth.tsx:15-18`), but the links discard it (`LoginPage.tsx:87-91`, `RegisterPage.tsx:101-105`); the destination then defaults to `/dashboard` (`auth-redirect.ts:1-16`). Smallest fix: pass `state={location.state}` on both cross-auth-page links and test register-from-protected-route returns to that route.
